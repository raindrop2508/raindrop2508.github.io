1. 问题现象 (The Problem)

在开发 Android 应用的“数据备份与恢复”功能时，我们需要在本地解压用户导入的 ZIP 压缩包。但测试时发现了一个诡异的现象：
部分 ZIP 包可以完美解压，而另一部分 ZIP 包在解压时会稳定引发崩溃，报错日志如下：

初步排查：

  • 报错的路径属于 App 的内部沙盒缓存目录（getCacheDir()），App 理应拥有绝对的读写权限。
  • 只有在 Windows 系统下打包的 ZIP 才会报错，而在 macOS 上打包的 ZIP 完全正常。

  • external_attr (转八进制): 0o100664（标准 UNIX 权限 -rw-rw-r--）

报错的压缩包 (Windows 打包)：

  • create_system: 0 (Windows/FAT)
  • external_attr (转八进制): 0o20（文件）或 0o30（目录）
  • 在 UNIX 权限体系（UNIX Permissions）中，0o30 对应的权限是 ----wx---，这意味着文件的所有者（Owner）没有任何读、写权限！

3. 根因分析：被忽视的 ZIP 底层属性 (Root Cause Analysis)

在 ZIP 文件格式规范（ZIP File Format Specification）中，每个被打包的文件或目录都会生成一个中央目录文件头（Central Directory File Header）。其中有两个关键的元数据字段导致了我们的 Bug：

  1. create_system (创建系统)：记录打包时的操作系统环境。常见值 0 为 Windows/DOS，3 为 UNIX/macOS。
  2. external_attr (外部文件属性)：记录文件打包时的属性状态。
    • 如果是 Windows (create_system == 0)，它存储的是 MS-DOS 属性（如只读、隐藏、归档）。
    • 如果是 UNIX (create_system == 3)，它的高 16 位存储的是标准的八进制 POSIX 权限（如 0o100664）。

致命的跨平台解析：
Android 底层是基于 Linux 的。当 zip4j 提取文件和目录时，它试图“尽职”地还原压缩包内记录的原始文件权限。但它忽略了检查 create_system 是否为 0 (Windows)。
它把 Windows 记录的 MS-DOS 属性，硬生生地按照 Linux 的 UNIX 权限去解析，结果歪打正着解析成了一个极低的权限（例如目录变成了 0o30，文件变成了 0o20）。

导致崩溃的完整连锁反应如下：

  1. zip4j 的 extractAll() 首先解压出了 images/ 目录。
  2. 它将错误解析的 0o30 权限强加给了该目录，导致该目录变得对 App 自身不可读写。
  3. 紧接着，zip4j 尝试将 images/cabinet_bg.png 写入该目录，内核层瞬间拦截，直接抛出 EACCES (Permission denied)！

4. 失败的尝试：事后补救 (The Pitfall)

一开始，我试图在 zf.extractAll() 执行完毕后，增加一个权限兜底方法，递归遍历解压出来的文件并调用 file.setReadable(true); file.setWritable(true);。

结果：依然崩溃。
因为异常是抛在 extractAll() 内部的。在解压子文件时，父目录权限已经被破坏导致抛出异常，代码直接中断，事后补救逻辑根本没有机会执行。

5. 完美解法：放弃自动解压，改用流式手动拷贝 (The Solution)

既然根源在于解压库错误地还原了文件权限，我们要做的就是切断权限属性的传递。放弃使用库自带的 extractAll，改用原生的 Java I/O 流进行手动拷贝。

在重构这段代码时，有一点必须保留：对 Charset（字符集）的支持。
Windows 下默认的压缩工具通常采用 GBK 编码来存储文件名，而 Android/macOS 默认使用 UTF-8。如果不做处理，包含中文的文件名（如 images/背景.png）解压后会变成乱码，导致后续业务逻辑报 FileNotFound。

最佳实践：采用“GBK 优先，UTF-8 兜底”的策略。

7. 进阶思考：未来适配 RAR 和 7Z 等格式该怎么办？ (Future Proofing)

在解决了 ZIP 的问题后，自然会引出一个疑问：如果后续业务需要支持 RAR 或 7Z 格式，是否需要针对每种格式去单独研究它的属性字段并做平台适配？

答案是：不需要。

虽然 RAR 的文件头包含 Host OS 字段，7Z 也通过复杂的 Attributes 来记录操作系统特征，但只要我们坚持上述修复方案中的核心防御策略：只取数据，不取属性（Stream-Only Extraction），就可以一劳永逸。

无论未来引入的是解压 RAR 的 junrar，还是解压 7Z 的 SevenZipJBinding，在编写解压代码时，绝对不要调用库提供的类似 extractAll(destPath) 的高级一键封装方法。

通用的解压防污染范式：

  1. 遍历压缩包条目（Entries）。
  2. 文件夹条目 -> 调用 new File(dir).mkdirs()。
  3. 文件条目 -> 获取该条目的 InputStream。
  4. 手动通过 FileOutputStream 写入数据。

因为只要你是通过 Java 原生的 FileOutputStream 和 File.mkdirs() 在 Android 沙盒中新建文件，底层内核（VFS）就会自动赋予当前 App 默认的最高读写权限。这就彻底屏蔽了任何压缩格式、在任何操作系统间穿梭所带来的权限异常风险。

总结

  1. 不要盲目信任第三方库的一键解压方法（如 extractAll），在跨平台（Windows -> Linux/Android）场景下，底层文件系统权限差异容易埋雷。
  2. 遇到 Permission denied 时，如果目录属于 App 自身沙盒，优先怀疑文件创建时的初始权限配置是否被外部输入（如 ZIP 属性）污染。
  3. 原生 I/O 流是隔离脏元数据最简单有效的方法。
